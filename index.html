What the platform does

T
Weighted supplier tiering

Eight configurable factors with thresholds and certification credits — every third party lands in a defensible Tier 1/2/3.

Q
Multi-domain IRQ engine

Conditional, tier-tailored questions with trigger logic that scopes risk domains and focus areas automatically. Fully editable in the Builder.

S
Risk-sensing screening

Cyber ratings, credit and sanctions connectors — integrate via API or upload the report. Flags evaluate automatically against thresholds.

A
Certification reliance

SOC 2 Type II controls map to individual DDQ questions; ISO 27001 covers focus areas. Tested controls are waived, report exceptions are flagged for testing — and for low-risk Tier 3 suppliers, a valid certificate is assessment enough.

C
Contractual obligations

An 18-clause cyber security schedule generated from the assessment — tier, scope, screening verdict and certification gaps — ready for the contract pack.

R
Risk & treatment tracking

Findings seed a risk register with owners, treatments, plans and due dates — rolled up across the portfolio so nothing falls through.

Inherent risk rating bands

How this platform works

  • Stage 1 — Profile & Tiering. Capture company info, vendor profile/demographics and operational factors. A weighted scoring model assigns Tier 1 / 2 / 3 (materiality, data sensitivity, criticality, substitutability, geography, fourth parties; certifications reduce the score).
  • Stage 2 — Inherent Risk (IRQ). The tier determines which IRQ questions are asked. Answers produce the inherent-risk rating and scope in DDQ domains via trigger logic.
  • Stage 3 — Screening. Gating rules decide if screening is needed (e.g. Tier 1 always; Tier 2 at Medium+; Tier 3 at High+). Pull or enter cyber ratings (BitSight, SecurityScorecard), credit ratings and sanctions/adverse-media results, then record an overall verdict.
  • Stage 4 — Due Diligence. Routing decides EDDQ (Tier 1, High/Critical rating, or adverse screening → all domains + mandatory evidence), DDQ (IRQ-scoped domains) or none. Track per-domain status and evidence.
  • Dashboard — the portfolio pipeline. Records are stored in this browser (localStorage); use Export all to back up.
  • Builder — edit every part of the model without touching code: tiering factors/weights/thresholds, gating, connectors, IRQ questions (incl. tier applicability), rules, domains, bands and EDDQ evidence.
  • Client workspaces — onboard clients and hand each one their own workspace page: only their ecosystem is visible, the Builder and other clients are hidden, exports are scoped to them, entry can be passcode-protected and exit guarded by an advisor PIN. Note: this is presentation-level segregation inside a local single-file app — for hard multi-tenant isolation with real logins, deploy the platform behind a backend.

Screening integrations

  • Each connector can call its API live (token stored only in this browser). Browsers block many of these calls from a local file (CORS) — if a fetch fails, use manual entry: enter the score/grade from the platform's portal or report. Flag thresholds auto-evaluate either way.
  • API tokens are never included in config or record exports.

Updating after the IRQ changes

  • Update from Excel (.xlsx) — re-import an updated IRQ spreadsheet. Rows are matched by Question ID; question text/options/guidance refresh while trigger rules, scores and tier settings are preserved for matching IDs.
  • Import / Export JSON — the entire platform configuration is one JSON document.

Excel format: first data columns in order — Q. No., Risk Area, Question, Configuration, Response Type, Response Options, User Guidance, Trigger Logic, Scoring.
